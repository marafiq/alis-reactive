⏺ Here's what onAllSuccess does:

  Runtime (pipeline.ts:46-51): After Promise.allSettled completes for ALL
  parallel requests, it runs onAllSuccess handlers. It fires regardless of
  whether individual requests succeeded or failed — it's a "when all are done"
  callback, not truly "all succeeded" (the name is misleading).

  DSL (ParallelBuilder.Response()):
  p.Parallel(
      r => r.Post("/api/save-a").Response(...),   // each branch has its own 
  chain
      r => r.Post("/api/save-b").Response(...)
  ).Response(r => r.OnSuccess(s =>                // ← this becomes onAllSuccess
  {
      s.Element("status").SetText("All done");
  }));

  Each branch already has its own complete request chain (gather, validate,
  onSuccess, onError, chained). onAllSuccess is the parallel-level response —
  commands that run after every branch finishes.

  Honestly though — looking at the runtime, it always fires (even if branches
  failed), so it's more like onAllSettled. And each branch already has its own
  onSuccess/onError via its RequestDescriptor. If you don't need a
  parallel-level "all done" signal, onAllSuccess is unnecessary.
