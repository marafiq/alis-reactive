Reactive is built on one idea: **express intent in C#, carry it as structured data, execute it in the browser.** Three layers, each with a clear boundary.

## Layer 1 — C# Fluent Builders

The developer writes fluent C# in `.cshtml` views. Every call builds a descriptor — a data structure that captures what should happen. Nothing executes. The builders are the API surface: triggers, mutations, conditions, HTTP, components.

The compiler checks model bindings, event arg types, and response types at build time. Rename a property → compiler error.

## Layer 2 — JSON Plan

`plan.Render()` serializes all descriptors to JSON. This JSON is the **contract** — the only thing that crosses from server to browser. The C# side never executes behavior. The JS side never invents behavior. Both agree on the plan's shape.

The plan is emitted as a `<script type="application/json">` element in the page. It is inert data until the runtime reads it.

## Layer 3 — JS Runtime

An ESM module loaded once via `_Layout.cshtml`. On page load, it discovers all plan elements, parses them, and executes the described behavior. The runtime is a **plan interpreter** — it reads structured commands and applies them via bracket notation on DOM elements and component instances.

The runtime does not know what a "checkbox" is or what "cascading dropdowns" means. It knows how to set a property, call a method, attach a listener, and make an HTTP request. The plan tells it which ones.

Every step in this flow is described in the plan. The runtime follows the instructions. If validation fails, the request never fires. If the server returns 400, the error handler runs. All decided by the plan, not by JavaScript logic.

## Bracket notation — the execution model

The runtime does not have a catalog of operations. It has two primitives:

| Mutation | Runtime executes |
|----------|-----------------|
| `set-prop` | `root[prop] = value` |
| `call` | `root[method](args)` |

`root` is resolved from the vendor — for native HTML it's the DOM element, for Syncfusion it's `el.ej2_instances[0]`. After that, bracket notation handles everything: setting text, adding classes, calling component methods. The plan carries the property/method name. The runtime applies it.

This is why adding a new component requires zero runtime changes — the C# vertical slice puts the right names in the plan. The runtime executes them verbatim.

## The architecture applied everywhere

Every feature follows the same three-layer pattern:

| Feature | C# captures | Plan carries | Runtime executes |
|---------|------------|-------------|-----------------|
| Events | `.Reactive(evt => evt.Changed)` | trigger + reaction | wires listener, runs commands |
| Mutations | `pipeline.Element("id").Show()` | target + mutation | bracket notation on DOM |
| Components | `pipeline.Component<T>(m => m.Prop)` | vendor + readExpr | resolves root, reads/writes |
| Conditions | `pipeline.When(...).Eq(val)` | guard + branches | evaluates, picks branch |
| HTTP | `pipeline.Post(url).Gather(...)` | request + handlers | fetches, routes response |

One idea, five features, zero JavaScript in your views.
