## 1. Discovery — `Scripts/root.ts`

The ESM entry point. Runs on every page load. Finds all `[data-reactive-plan]` elements, parses their JSON, composes partial plans, and calls `boot()` for each.

No global state. No `window.alis` (with one exception: `window.alis.confirm` is exposed so the `confirm` guard can show a browser confirmation dialog — this is the only global surface). The layout loads the runtime once via `<script type="module" src>`. Discovery handles the rest.

## 2. Boot — `Scripts/lifecycle/boot.ts`

Orchestrates the plan lifecycle in three steps:

```d2
direction: right

enrich: "enrichment.ts\nResolve component refs\nto IDs, vendors, readExprs" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
validation: "walk-reactions.ts\nFind validation descriptors\nWire live-clear listeners" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
wire: "wireEntries()\nTwo-phase trigger wiring" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}

enrich -> validation -> wire
```

**Enrichment** runs first: it walks all plan entries and resolves abstract component references — gather placeholders and validation field refs — to concrete IDs, vendors, and readExprs from the ComponentsMap. After enrichment, every entry is fully resolved and ready for trigger wiring. No late-binding remains.

**Two-phase wiring:** Non-dom-ready listeners attach first (Phase 1). Then dom-ready reactions execute (Phase 2). This guarantees dispatched events land on existing listeners — regardless of declaration order in the view.

Also handles `mergePlan()` for partial views loaded via AJAX — same enrichment + wiring, applied to the incoming entries.

## 3. Trigger wiring — `Scripts/execution/trigger.ts`

Attaches event listeners based on `trigger.kind`. Each listener creates an execution context and calls `executeReaction()`.

| Trigger kind | What it wires | Context created |
|-------------|---------------|-----------------|
| `dom-ready` | Immediate execution (or `DOMContentLoaded`) | `{ components }` |
| `custom-event` | `document.addEventListener(eventName)` | `{ evt: detail, components }` |
| `component-event` | `resolveRoot(el, vendor).addEventListener(jsEvent)` | `{ evt: payload, components }` |

For component events, `resolveRoot()` from `Scripts/resolution/component.ts` determines whether to listen on the DOM element (native) or the Syncfusion instance (fusion).

All listeners use `AbortSignal` for clean teardown during plan merges and test resets.

## 4. User checks the checkbox

The `change` event fires on the native checkbox element. The listener wired in step 3 creates `{ evt: { checked: true }, components }` and calls `executeReaction()`.

## 5. Reaction dispatch — `Scripts/execution/execute.ts`

Switches on `reaction.kind`:

| Kind | Behavior |
|------|----------|
| `sequential` | Executes each command in order |
| `conditional` | Executes commands that pass conditions. Supports if/elseif/else |
| `http` | Delegates to `Scripts/execution/pipeline.ts` — validate, gather, fetch, route response |
| `parallel-http` | `Promise.all` on multiple requests, then `OnAllSettled` commands |

Fully async — any branch may contain an HTTP reaction. The execution context (`evt`, `components`) threads through recursively.

## 6. Command dispatch — `Scripts/execution/commands.ts`

Switches on `command.kind`:

| Kind | Handler |
|------|---------|
| `dispatch` | Fires `new CustomEvent(name, { detail })` on `document` |
| `mutate-element` | Delegates to `Scripts/execution/element.ts` |
| `mutate-event` | Modifies `ctx.evt` properties (used by filtering args) |
| `validation-errors` | Delegates to `Scripts/validation/error-display.ts` |
| `into` | Injects response HTML, merges any embedded plans |

Supports optional per-command `when` guards — evaluated before the command executes. If the guard fails, the command is skipped but the pipeline continues.

## 7. Condition evaluation — `Scripts/conditions/conditions.ts`

Evaluates guard expressions. The runtime reads `guard.kind` and follows one of five paths:

`value` guards resolve their source via `resolver.ts` (event path or component read), coerce both sides, then apply the operator. `all`/`any`/`not` are recursive — they compose inner guards. `confirm` is the only async guard — it pauses the pipeline and waits for user response.

## 8. Element mutation — `Scripts/execution/element.ts`

The execution endpoint. Looks up the DOM element by ID, resolves the vendor root, and applies one of two mutation primitives:

```d2
direction: right

lookup: "getElementById(target)" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
resolve: "resolveRoot(el, vendor)\nScripts/resolution/component.ts" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
setprop: "set-prop\nroot[prop] = val" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
call: "call\nroot[method](args)" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}

lookup -> resolve -> setprop
resolve -> call
```

Method arguments are resolved individually — each `MethodArg` is either a `literal` (pass-through) or a `source` (resolved via `resolver.ts` with optional coercion).

`resolveRoot()` in `Scripts/resolution/component.ts` is the **only function** in the runtime that knows about `ej2_instances`. Everything upstream and downstream is vendor-agnostic.

## 9. Value resolution — `Scripts/resolution/resolver.ts`

Dispatches by `source.kind`:

| Source kind | Resolution |
|------------|-----------|
| `event` | `walk(ctx, "evt.address.city")` — dot-path into execution context |
| `component` | `evalRead(id, vendor, readExpr)` — vendor root + dot-path walk |

Both converge on one primitive.

## 10. walk — `Scripts/core/walk.ts`

This is the most important function in the runtime. Every read, every source resolution, every condition evaluation flows through it. 7 lines, zero side effects:

```d2
direction: right

path: "path = 'evt.address.city'" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
split: "split('.')\n['evt', 'address', 'city']" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
step1: "root['evt']\n→ event payload" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
step2: "payload['address']\n→ address object" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
step3: "address['city']\n→ 'Portland'" {
  shape: rectangle
  style.fill: "#d4edda"
  style.stroke: "#34a853"
  style.font-size: 16
}

path -> split -> step1 -> step2 -> step3
```

Split the path on dots. Walk each segment with bracket notation: `current = current[part]`. If any segment is null, return `undefined`. That's it.

This is how the C# expression `x => x.Address.City` reaches the browser value. `ExpressionPathHelper` converts it to `"evt.address.city"` at build time. The plan carries the string. `walk()` traverses it at runtime. The two layers never meet — the dot-path is the contract between them.

`walk` is used by:
- **resolver.ts** — walk event payloads (`ctx.evt.address.city`)
- **component.ts** — walk component readExpr (`root.value`, `root.checked`)
- **conditions.ts** — resolve guard source values before comparing
- **gather.ts** — read component values for HTTP request bodies
