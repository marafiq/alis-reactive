A native checkbox is `el.checked`. A Syncfusion date picker is `el.ej2_instances[0].value`. The runtime handles both through one code path. This page explains how.

For the general execution model (bracket notation, command dispatch), see [Runtime](/architecture/runtime/).

## Three metadata fields

Each component declares three things in C#. The plan carries them. The runtime uses them.

| Field | Purpose | Example |
|-------|---------|---------|
| **Vendor** | Which root object to resolve | `"native"` or `"fusion"` |
| **ReadExpr** | Which property to read | `"value"`, `"checked"` |
| **ComponentType** | What kind of component | `"textbox"`, `"datepicker"` |

When you write `Html.InputField(plan, m => m.DueDate).FusionDatePicker(...)`, the builder registers `{ vendor: "fusion", readExpr: "value", componentType: "datepicker" }` in the plan's component registry.

## Vendor resolution — one function, any vendor

```d2
direction: right

element: "DOM Element" {
  shape: rectangle
  style.fill: "#f0f0f0"
  style.stroke: "#666"
  style.font-size: 18
}

native_root: "native\nroot = element" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 18
}

fusion_root: "fusion\nroot = el.ej2_instances[0]" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 18
}

bracket: "root[prop] = val\nroot[method](args)\nSame code path" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 18
}

element -> native_root: vendor {
  style.stroke: "#4285f4"
}
element -> fusion_root: vendor {
  style.stroke: "#f9ab00"
}
native_root -> bracket {
  style.stroke: "#34a853"
}
fusion_root -> bracket {
  style.stroke: "#34a853"
}
```

`component.ts` is the module that resolves vendor roots (`ej2_instances`). `trigger.ts` also has vendor-aware logic for constructing event detail objects — native components read `readExpr` from the element, while fusion components pass the event object directly. After resolving the root, every operation flows through bracket notation — the same two primitives (`set-prop`, `call`) described in [Runtime](/architecture/runtime/).

## Every interaction, one pattern

| Interaction | Plan carries | Runtime does |
|-------------|-------------|-------------|
| **Read** (gather, conditions) | vendor + readExpr | `walk(resolveRoot(el, vendor), readExpr)` |
| **Write** (SetValue, SetChecked) | vendor + prop | `resolveRoot(el, vendor)[prop] = val` |
| **Call** (DataBind, ShowPopup) | vendor + method | `resolveRoot(el, vendor)[method]()` |
| **Wire event** (.Reactive) | vendor + jsEvent | `resolveRoot(el, vendor).addEventListener(jsEvent)` |

## Adding a new component — zero runtime changes

```d2
direction: right

csharp: "New C# vertical slice\nVendor + ReadExpr" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 18
}

plan: "Plan carries metadata\nSame shape as every component" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 18
}

runtime: "Runtime unchanged\nAlready knows the vendor\nAlready knows bracket notation" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 18
}

csharp -> plan: metadata {
  style.stroke: "#4285f4"
}
plan -> runtime: same execution {
  style.stroke: "#34a853"
}
```

No new `if` branches, no new switch cases, no runtime deployment. The component declares its vendor and readExpr. The runtime resolves and executes.
