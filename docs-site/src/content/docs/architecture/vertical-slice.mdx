Every component in Reactive is a **vertical slice** — a self-contained set of files that owns everything about that component: its type metadata, events, mutations, builder, and view integration. No shared base classes for behavior. No cross-slice dependencies.

## The 7-file pattern

Each component follows the same structure. Here's NativeTextBox as a concrete example:

```
NativeTextBox/
├── NativeTextBox.cs                  component class — Vendor + ReadExpr
├── NativeTextBoxEvents.cs            event registry — Changed, Focus, Blur
├── Events/
│   └── NativeTextBoxOnChanged.cs     event args — { string? Value }
├── NativeTextBoxExtensions.cs        mutation extensions — SetValue, FocusIn, Value
├── NativeTextBoxReactiveExtensions.cs .Reactive() wiring
├── NativeTextBoxBuilder.cs           renders HTML, config methods (Placeholder, Type, CssClass)
└── NativeTextBoxHtmlExtensions.cs    Html.InputField().NativeTextBox() factory
```

## What each file does

### 1. Component class — the phantom type

A `sealed class` with a parameterless constructor. Implements `IInputComponent` (or `IAppLevelComponent`). Declares `Vendor` and `ReadExpr` as instance properties. Has zero state — it exists only as a type parameter to constrain extension methods.

### 2. Events registry — what the component can fire

A singleton class with `TypedEventDescriptor<TArgs>` properties. The `.Reactive()` lambda `evt => evt.Changed` resolves to this descriptor, which carries the JS event name and the typed args class.

### 3. Event args — typed payload properties

A class with properties like `Value : string?`, `Checked : bool?`, `IsInteracted : bool`. The compiler enforces which properties exist on `args` inside the `.Reactive()` lambda. Filtering args may also have extension methods (`PreventDefault`, `UpdateData`).

### 4. Mutation extensions — what you can do in the pipeline

Static extension methods on `ComponentRef<TComponent, TModel>`. Each method creates a structured `Mutation` (SetPropMutation or CallMutation) and calls `ref.Emit()` to add it to the pipeline. `.Value()` returns a `TypedComponentSource<TProp>` for conditions.

### 5. Reactive extensions — wiring events to pipelines

A static extension method on the builder that takes `plan`, an event selector, and a pipeline lambda. Creates a `ComponentEventTrigger` with the component's vendor, jsEvent, and bindingPath, then adds the entry to the plan.

### 6. Builder — renders HTML and accepts configuration

Implements `IHtmlContent`. Configuration methods (`Placeholder()`, `CssClass()`, `Items()`) are fluent. `WriteTo()` renders the HTML element. The constructor is internal — devs use the Html factory.

### 7. Html factory — the entry point

Extension method on `InputFieldSetup<TModel, TProp>`. Creates the builder, registers the component in `plan.ComponentsMap`, and calls `setup.Render(builder)` to emit the HTML with label and validation slot.

## Why vertical slices?

**Isolation.** Each slice owns its behavior completely. Adding FusionTimePicker doesn't touch NativeTextBox. Changing NativeCheckBox's events doesn't affect FusionDropDownList.

**Zero runtime changes.** The slice declares Vendor and ReadExpr. The runtime already knows how to resolve vendors and walk readExprs. The plan carries the metadata. No new switch cases, no new if branches.

**Duplication is intentional.** Each slice has its own extensions file, its own reactive extensions, its own builder. The code looks similar across slices — that's by design. Shared abstractions would couple slices and make each one harder to understand in isolation.
