Two-phase boot is an architectural invariant. It solves a specific problem that emerges whenever event chains span multiple plan entries.

## The problem

Consider a plan with two entries:

1. **On dom-ready**: dispatch `"resident-loaded"` with payload `{ name: "Margaret Chen" }`
2. **On custom-event `"resident-loaded"`**: set the text of `#resident-name` to the payload's `name`

Entry order in the plan is determined by the order `Html.On()` calls appear in the view. But the developer shouldn't need to think about ordering. Both entries are declarations of behavior --- "when this happens, do that" --- and they should work regardless of which appears first.

If the runtime processes entries sequentially --- wire trigger, then execute --- the dom-ready entry fires immediately. It dispatches `"resident-loaded"`. But the custom-event listener hasn't been wired yet. The event vanishes. The resident's name never appears.

This is not a theoretical concern. Event chains are the primary composition mechanism in Reactive. A dom-ready trigger loads data from the server. The response handler dispatches an event with the data. Downstream listeners update the UI. If the listeners aren't ready, the chain breaks silently.

## The solution

The runtime boots in two phases:

**Phase 1** --- Wire all non-dom-ready triggers. Custom-event listeners and component-event listeners are attached to the document. Nothing executes yet.

**Phase 2** --- Execute dom-ready triggers. These may dispatch events, which phase 1 listeners are now ready to catch.

## The implementation

The boot function collects dom-ready entries into a deferred list, wires everything else first, then processes the deferred entries:

```typescript
function wireEntries(
  entries: Entry[],
  components: Record<string, ComponentEntry>,
  signal?: AbortSignal
): void {
  // Phase 1: wire all non-dom-ready listeners
  const deferred: Entry[] = [];
  for (const entry of entries) {
    if (entry.trigger.kind === "dom-ready") {
      deferred.push(entry);
    } else {
      wireTrigger(entry.trigger, entry.reaction, components, signal);
    }
  }

  // Phase 2: execute dom-ready triggers
  for (const entry of deferred) {
    wireTrigger(entry.trigger, entry.reaction, components, signal);
  }
}
```

`wireTrigger` handles dom-ready entries by executing their reaction immediately (there's nothing to "listen" for --- the DOM is already ready). For custom-event and component-event entries, it attaches an event listener that will execute the reaction when the event fires.

The separation is deliberate: phase 1 entries register callbacks. Phase 2 entries invoke callbacks. By the time phase 2 runs, every callback is in place.

## Event chains of any depth

Two-phase boot doesn't just handle one hop. It handles chains of arbitrary depth:

```
dom-ready
  → dispatch("data-loaded")
    → custom-event("data-loaded")
      → dispatch("ui-ready")
        → custom-event("ui-ready")
          → mutate elements
```

This works because `dispatch` fires a synchronous `CustomEvent` on `document`. The listener wired in phase 1 receives it immediately. If that listener's reaction also dispatches, the next listener (also wired in phase 1) receives that event immediately too. The entire chain completes synchronously within the phase 2 execution.

## What about async reactions?

HTTP reactions (`kind: "http"`) are asynchronous --- the fetch completes later. But the event chain still works. The HTTP reaction's response handler dispatches an event when the response arrives. By that time, all listeners from phase 1 are long since wired. Two-phase boot guarantees listener availability at boot time; it doesn't need to guarantee it during async continuations because the listeners persist for the lifetime of the page.

## Verification at all three layers

This invariant is too important to verify at just one level.

**C# unit tests** verify that the plan structure is correct --- dom-ready triggers and custom-event triggers serialize properly, and a plan can contain both.

**TS unit tests** (`when-dom-ready-chains-into-custom-event.test.ts`) create a plan where dom-ready dispatches an event and a custom-event listener mutates the DOM. The test calls `boot()` directly and asserts that the DOM mutation occurred. If two-phase boot breaks, this test fails.

**Playwright tests** (`WhenEventChainFires.cs`) navigate to a real page in the sandbox app where a three-hop dispatch chain runs on page load. The test waits for trace messages in the browser console and asserts that all hops completed in chronological order. If the runtime script changes and breaks the boot order, the browser test catches it.

Three layers, one invariant: **listeners are wired before dispatches fire**. Every commit verifies this.
