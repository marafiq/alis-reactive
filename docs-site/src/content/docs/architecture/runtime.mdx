The runtime is a plan interpreter. It reads the JSON plan that C# builders produce and executes the described behavior. It does not invent behavior — it follows instructions.

## Module structure

```
Scripts/
├── root.ts                          entry point — discovers [data-reactive-plan], calls boot()
├── lifecycle/
│   ├── boot.ts                      two-phase boot — wire listeners, then execute DomReady
│   ├── enrichment.ts                fills validation field metadata from ComponentsMap
│   ├── merge-plan.ts                composes partial plans (ResolvePlan merges into parent)
│   └── walk-reactions.ts            traverses reaction tree for validation + live-clear wiring
├── execution/
│   ├── execute.ts                   reaction dispatcher — sequential, conditional, http, parallel
│   ├── commands.ts                  command dispatcher — dispatch, mutate-element, mutate-event, into
│   ├── element.ts                   bracket notation — root[prop] = val, root[method](args)
│   ├── trigger.ts                   wires dom-ready, custom-event, component-event listeners
│   ├── gather.ts                    collects component + event values for HTTP requests
│   ├── http.ts                      builds and executes fetch(), routes responses
│   ├── pipeline.ts                  orchestrates: pre-fetch → validate → request → parallel
│   └── inject.ts                    injects HTML into containers, merges embedded plans
├── resolution/
│   ├── resolver.ts                  walks BindSource — event paths or component reads
│   └── component.ts                 vendor root resolution (see Component Model)
├── conditions/                      evaluates guards — value, all, any, not, confirm
├── validation/                      client-side validation from plan descriptors
├── components/                      component-specific side effects (drawer, loader, confirm)
├── core/                            walk.ts, coerce.ts, trace.ts, assert-never.ts
└── types/                           TypeScript interfaces mirroring the JSON plan schema
```

## Boot — `root.ts` → `boot.ts`

`root.ts` runs on every page load. It finds all `[data-reactive-plan]` elements, parses their JSON, composes partial plans via `composeInitialPlans()`, and calls `boot()` for each. No global state. No `window.alis` (one exception: `window.alis.confirm` for the confirm guard dialog).

`boot()` orchestrates three steps:

```d2
direction: right

enrich: "enrichment.ts\nFill validation fields\nfrom ComponentsMap" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
validation: "walk-reactions.ts\nFind validation descriptors\nWire live-clear listeners" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
wire: "wireEntries()\nTwo-phase trigger wiring" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}

enrich -> validation -> wire
```

**Enrichment** walks validation descriptors and fills in `fieldId`, `vendor`, and `readExpr` on each field from the ComponentsMap. This is the only runtime enrichment — gather placeholders and component trigger refs are already resolved at C# render time.

**Two-phase boot** guarantees event chains work regardless of declaration order. Phase 1 wires all CustomEvent and component-event listeners. Phase 2 runs DomReady reactions — which may dispatch events that Phase 1 listeners catch. Without this, a DomReady dispatch would fire before the listener exists and vanish silently.

At this point, the page is live. Listeners are attached. The runtime is waiting.

## When a trigger fires — `trigger.ts` → `execute.ts` → `element.ts`

The user checks a checkbox. The `change` listener fires. The runtime makes three decisions — each one narrowing scope until the DOM changes.

### Trigger wiring — `trigger.ts`

Attaches listeners based on `trigger.kind`. Each creates an execution context and calls `executeReaction()`.

| Trigger kind | What it wires | Context |
|-------------|---------------|---------|
| `dom-ready` | Immediate execution (or `DOMContentLoaded`) | `{ components }` |
| `custom-event` | `document.addEventListener(eventName)` | `{ evt: detail, components }` |
| `component-event` | `resolveRoot(el, vendor).addEventListener(jsEvent)` | `{ evt: payload, components }` |

For component events, `trigger.ts` contains vendor-specific logic: native components read `readExpr` from the element and package it into the detail, fusion components pass the event object directly. All listeners use `AbortSignal` for clean teardown.

### Decision 1 — Reaction kind (`execute.ts`)

| Kind | Behavior | Sync/Async |
|------|----------|-----------|
| `sequential` | Execute each command in order | Sync |
| `conditional` | Evaluate guards, pick matching branch (if/elseif/else) | Async (guards) |
| `http` | Validate → gather → fetch → route response | Async (fetch) |
| `parallel-http` | `Promise.all` on requests, then `OnAllSettled` | Async |

The tree is recursive — a conditional branch can contain an HTTP reaction, and vice versa — but at each node the dispatch is a flat switch on `kind`.

### Decision 2 — Command kind (`commands.ts`)

| Kind | Module | What it does |
|------|--------|-------------|
| `dispatch` | inline | Fires `CustomEvent` on `document` |
| `mutate-element` | `element.ts` | Bracket notation on DOM element or component root |
| `mutate-event` | inline | Modifies `ctx.evt` (e.g., `preventDefaultAction = true`) |
| `validation-errors` | `validation/` | Routes server-side errors to form fields |
| `into` | `inject.ts` | Injects response HTML, merges any embedded plans |

Each command can carry an optional `when` guard — if it fails, the command is skipped but the pipeline continues.

### Decision 3 — Mutation kind (`element.ts` / `commands.ts`)

Two primitives, three targets:

- **`set-prop`** — `root[prop] = val`
- **`call`** — `root[method](args)`

| Target | Module | How root is resolved |
|--------|--------|---------------------|
| DOM element | `element.ts` | `getElementById(target)` |
| Component | `element.ts` | `resolveRoot(el, vendor)` |
| Event args | `commands.ts` | `ctx.evt` |

The plan carries every name as a string. The runtime applies it via bracket notation — no catalog, no switch on method names.

## Value resolution — `resolver.ts` → `walk.ts`

Commands and conditions need dynamic values. `resolver.ts` dispatches by `source.kind`:

| Source | Resolution path |
|--------|---------------|
| `event` | `walk(ctx, "evt.address.city")` |
| `component` | `resolveRoot(el, vendor)` → `walk(root, readExpr)` via `evalRead()` in `component.ts` |

Both converge on `walk()` — the most important function in the runtime:

```d2
direction: right

path: "'evt.address.city'" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
split: "split('.')\n['evt','address','city']" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
step1: "ctx['evt']" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
step2: "evt['address']" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
step3: "addr['city']\n→ 'Portland'" {
  shape: rectangle
  style.fill: "#d4edda"
  style.stroke: "#34a853"
  style.font-size: 16
}

path -> split -> step1 -> step2 -> step3
```

Split on dots, walk each segment with bracket notation. If any segment is null, return `undefined`. This is how the C# expression `x => x.Address.City` reaches the browser value — `ExpressionPathHelper` converts it to `"evt.address.city"` at build time, `walk()` traverses it at runtime.

`value` guards resolve via `resolver.ts`, coerce both sides, apply the operator (20+ operators). `all`/`any`/`not` compose recursively. `confirm` is the only async guard — shows a browser dialog via `window.alis.confirm`.
