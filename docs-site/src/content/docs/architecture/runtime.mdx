The runtime is a plan interpreter. It reads the JSON plan that C# builders produce and executes the described behavior. It does not invent behavior ‚Äî it follows instructions.

## Module structure

```
Scripts/
‚îú‚îÄ‚îÄ root.ts                          entry point ‚Äî discovers [data-reactive-plan], calls boot()
‚îú‚îÄ‚îÄ lifecycle/
‚îÇ   ‚îú‚îÄ‚îÄ boot.ts                      two-phase boot ‚Äî wire listeners, then execute DomReady
‚îÇ   ‚îú‚îÄ‚îÄ enrichment.ts                fills validation field metadata from ComponentsMap
‚îÇ   ‚îú‚îÄ‚îÄ merge-plan.ts                composes partial plans (ResolvePlan merges into parent)
‚îÇ   ‚îî‚îÄ‚îÄ walk-reactions.ts            traverses reaction tree for validation + live-clear wiring
‚îú‚îÄ‚îÄ execution/
‚îÇ   ‚îú‚îÄ‚îÄ execute.ts                   reaction dispatcher ‚Äî sequential, conditional, http, parallel
```

## Boot ‚Äî `root.ts` ‚Üí `boot.ts`

`root.ts` runs on every page load. It finds all `[data-reactive-plan]` elements, parses their JSON, composes partial plans via `composeInitialPlans()`, and calls `boot()` for each. No global state. No `window.alis` (one exception: `window.alis.confirm` for the confirm guard dialog).

`boot()` orchestrates three steps:

```d2
direction: right

enrich: "enrichment.ts\nFill validation fields\nfrom ComponentsMap" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
validation: "walk-reactions.ts\nFind validation descriptors\nWire live-clear listeners" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
wire: "wireEntries()\nTwo-phase trigger wiring" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}

enrich -> validation -> wire
```

**Enrichment** walks validation descriptors and fills in `fieldId`, `vendor`, `readExpr`, and `coerceAs` on each field from the ComponentsMap. The `coerceAs` field carries the type inferred from `typeof(TProp)` at C# registration time ‚Äî used by the condition reader for typed comparison (e.g., Unix ms for dates). This is the only runtime enrichment ‚Äî gather placeholders and component trigger refs are already resolved at C# render time.

**Two-phase boot** guarantees event chains work regardless of declaration order. Phase 1 wires all CustomEvent and component-event listeners. Phase 2 runs DomReady reactions ‚Äî which may dispatch events that Phase 1 listeners catch. Without this, a DomReady dispatch would fire before the listener exists and vanish silently.

At this point, the page is live. Listeners are attached. The runtime is waiting.

## When a trigger fires ‚Äî `trigger.ts` ‚Üí `execute.ts` ‚Üí `element.ts`

The user checks a checkbox. The `change` listener fires. The runtime makes three decisions ‚Äî each one narrowing scope until the DOM changes.

### Trigger wiring ‚Äî `trigger.ts`

Attaches listeners based on `trigger.kind`. Each creates an execution context and calls `executeReaction()`.

| Trigger kind | What it wires | Context |
|-------------|---------------|---------|
| `dom-ready` | Immediate execution (or `DOMContentLoaded`) | `{ components }` |
| `custom-event` | `document.addEventListener(eventName)` | `{ evt: detail, components }` |
| `component-event` | `resolveRoot(el, vendor).addEventListener(jsEvent)` | `{ evt: payload, components }` |

For component events, `trigger.ts` contains vendor-specific logic: native components read `readExpr` from the element and package it into the detail, fusion components pass the event object directly. All listeners use `AbortSignal` for clean teardown.

### Decision 1 ‚Äî Reaction kind (`execute.ts`)

| Kind | Behavior | Sync/Async |
|------|----------|-----------|
| `sequential` | Execute each command in order | Sync |
| `conditional` | Evaluate guards, pick matching branch (if/elseif/else) | Async (guards) |
| `http` | Validate ‚Üí gather ‚Üí fetch ‚Üí route response | Async (fetch) |
| `parallel-http` | `Promise.all` on requests, then `OnAllSettled` | Async |

The tree is recursive ‚Äî a conditional branch can contain an HTTP reaction, and vice versa ‚Äî but at each node the dispatch is a flat switch on `kind`.

### Decision 2 ‚Äî Command kind (`commands.ts`)

| Kind | Module | What it does |
|------|--------|-------------|
| `dispatch` | inline | Fires `CustomEvent` on `document` |
| `mutate-element` | `element.ts` | Bracket notation on DOM element or component root |
| `mutate-event` | inline | Modifies `ctx.evt` (e.g., `preventDefaultAction = true`) |
| `validation-errors` | `validation/` | Routes server-side errors to form fields |
| `into` | `inject.ts` | Injects response HTML, merges any embedded plans |

Each command can carry an optional `when` guard ‚Äî if it fails, the command is skipped but the pipeline continues.

### Decision 3 ‚Äî Mutation kind (`element.ts` / `commands.ts`)

Two primitives, three targets:

- **`set-prop`** ‚Äî `root[prop] = val`
- **`call`** ‚Äî `root[method](args)`

| Target | Module | How root is resolved |
|--------|--------|---------------------|
| DOM element | `element.ts` | `getElementById(target)` |
| Component | `element.ts` | `resolveRoot(el, vendor)` |
| Event args | `commands.ts` | `ctx.evt` |

The plan carries every name as a string. The runtime applies it via bracket notation ‚Äî no catalog, no switch on method names.

## Value resolution ‚Äî `resolver.ts` ‚Üí `walk.ts`

Commands and conditions need dynamic values. `resolver.ts` dispatches by `source.kind`:

| Source | Resolution path |
|--------|---------------|
| `event` | `walk(ctx, "evt.address.city")` |
| `component` | `resolveRoot(el, vendor)` ‚Üí `walk(root, readExpr)` via `evalRead()` in `component.ts` |

Both converge on `walk()` ‚Äî the most important function in the runtime:

```d2
direction: right

path: "'evt.address.city'" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 16
}
split: "split('.')\n['evt','address','city']" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 16
}
step1: "ctx['evt']" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
step2: "evt['address']" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 16
}
step3: "addr['city']\n‚Üí 'Portland'" {
  shape: rectangle
  style.fill: "#d4edda"
  style.stroke: "#34a853"
  style.font-size: 16
}

path -> split -> step1 -> step2 -> step3
```

Split on dots, walk each segment with bracket notation. If any segment is null, return `undefined`. This is how the C# expression `x => x.Address.City` reaches the browser value ‚Äî `ExpressionPathHelper` converts it to `"evt.address.city"` at build time, `walk()` traverses it at runtime.

## Coercion ‚Äî `coerce.ts`

The coerce module serves two distinct operations that share the type system but never mix:

| Operation | Function | Purpose | Used by |
|-----------|----------|---------|---------|
| **Comparison** | `coerce(raw, type)` | Convert to typed value for operators | Conditions, Mutations, Validation comparison rules |
| **Serialization** | `toString(raw)` | Convert to string for HTTP transport | Gather (FormData, GET). JSON uses `JSON.stringify` on raw values. |

Every coerce function returns `CoerceResult<T>` ‚Äî a discriminated union `{ ok: true, value: T } | { ok: false, error: string }`. Functions never throw. Callers decide how to handle `Err`:

| Consumer | On Err | Why |
|----------|--------|-----|
| Conditions | Guard ‚Üí false, else branch runs | User interaction must not crash |
| Validation | Rule fails, shows error message | User sees field error, not app crash |
| Mutations | `coerceOrThrow()` ‚Äî throws with stack trace | Developer error in plan needs debugging |
| Gather | Log warning, skip field | Don't crash entire HTTP request |

Supported coercion types (inferred from C# `typeof(TProp)` via `CoercionTypes.InferFromType`):

| CoercionType | C# types | Comparison result | Serialization result |
|-------------|----------|-------------------|---------------------|
| `"string"` | `string`, `Enum` | identity | identity |
| `"number"` | `int`, `decimal`, `double`, ... | number | `"4250"` |
| `"boolean"` | `bool` | `true`/`false` | `"true"`/`"false"` |
| `"date"` | `DateTime`, `DateTimeOffset`, `DateOnly` | Unix ms (getTime) | ISO 8601 (toISOString) |
| `"array"` | `T[]`, `List<T>`, ... | identity | JSON.stringify |
| `"raw"` | everything else | pass-through | Err (plain objects) |

The plan carries `coerceAs` on `ComponentEntry` (from `typeof(TProp)`), `ValueGuard` (from conditions), `SetPropMutation` (from mutations), and `ValidationRule` (from FluentValidation). The runtime never infers or guesses a coercion type ‚Äî it reads the plan.
