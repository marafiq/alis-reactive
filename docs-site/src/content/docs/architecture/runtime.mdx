The runtime is a plan interpreter. It reads the JSON plan that C# builders produce and executes the described behavior. It does not invent behavior — it follows instructions.

## Module structure

```
Scripts/
├── root.ts                          entry point — discovers [data-reactive-plan], calls boot()
├── lifecycle/                       boot, merge, enrichment
│   ├── boot.ts                      two-phase boot — wire listeners, then execute DomReady
│   ├── enrichment.ts                resolves component refs to IDs, vendors, readExprs
│   ├── merge-plan.ts                composes partial plans (ResolvePlan merges into parent)
│   └── walk-reactions.ts            traverses reaction tree for validation + live-clear wiring
├── execution/                       where behavior happens
│   ├── execute.ts                   reaction dispatcher — sequential, conditional, http, parallel
│   ├── commands.ts                  command dispatcher — dispatch, mutate-element, mutate-event, into
│   ├── element.ts                   bracket notation — root[prop] = val, root[method](args)
│   ├── trigger.ts                   wires dom-ready, custom-event, component-event listeners
│   ├── gather.ts                    collects component + event values for HTTP requests
│   ├── http.ts                      builds and executes fetch(), routes responses
│   ├── pipeline.ts                  orchestrates: pre-fetch → validate → request → parallel
│   └── inject.ts                    injects HTML into containers, merges embedded plans
├── resolution/                      where values come from
│   ├── resolver.ts                  walks BindSource — event paths or component reads
│   └── component.ts                 vendor root resolution (see Component Model)
├── conditions/                      evaluates guards — value, all, any, not, confirm
├── validation/                      client-side validation from plan descriptors
├── components/                      component-specific side effects (drawer, loader, confirm)
├── core/                            walk.ts, coerce.ts, trace.ts, assert-never.ts
└── types/                           TypeScript interfaces mirroring the JSON plan schema
```

```d2
  style.stroke: "#4285f4"

```

**Two-phase boot** guarantees that event chains work regardless of declaration order. Phase 1 wires all CustomEvent and component-event listeners. Phase 2 runs DomReady reactions — which may dispatch events that Phase 1 listeners catch. Without this separation, a DomReady dispatch would fire before the listener exists and the event would vanish silently. Chains of any depth work because `dispatch` fires synchronous `CustomEvent`s on `document`.

At this point, the page is live. Listeners are attached. The runtime is waiting.

## When a trigger fires

The user checks a checkbox. The `change` listener fires. Now what?

Read the execution from left to right — each step narrows the scope until the DOM changes. The runtime makes exactly three decisions: what kind of reaction, what kind of command, what kind of mutation.

### Decision 1 — Reaction kind (`execute.ts`)

The trigger carries a reaction. The runtime reads `reaction.kind` and picks the strategy:

- **`sequential`** — walk the command list, execute each one in order
- **`conditional`** — executes commands that pass conditions. Supports if/elseif/else
- **`http`** — validate the form, gather component values, fetch, route the response to handlers
- **`parallel-http`** — fire multiple requests concurrently via `Promise.all`, then run `OnAllSettled`

A conditional branch can contain an HTTP reaction. An HTTP response handler can contain a conditional. The tree is recursive, but at each node the decision is a flat switch on `kind`.

### Decision 2 — Command kind (`commands.ts`)

Inside the chosen reaction, the runtime iterates the command list. Each command declares its `kind`:

- **`dispatch`** — fires a `CustomEvent` on `document` (other listeners may catch it)
- **`mutate-element`** — single command for both `pipeline.Element()` and `pipeline.Component<T>()`. When `vendor` is present, resolves the component root first. Same command, same code path (next decision)
- **`mutate-event`** — modifies the event args in the execution context
- **`validation-errors`** — routes server-side errors to form fields
- **`into`** — injects HTTP response HTML into a container element

Each command can carry an optional `when` guard. If the guard fails, that command is skipped — but the pipeline continues with the next command. This is how per-command conditions work without wrapping everything in When/Then.

### Decision 3 — Mutation kind (`element.ts` and `commands.ts`)

Two mutation primitives, applied to three targets:

- **`set-prop`** — `root[prop] = val` — handles SetText, SetValue, SetChecked
- **`call`** — `root[method](args)` — handles AddClass, DataBind, ShowPopup

DOM elements and components are mutated by `element.ts` (with vendor root resolution). Event args are mutated by `commands.ts` directly. Reads flow through a separate path: `resolver.ts` → `walk.ts`.

The plan carries every name as a string. The runtime applies it via bracket notation — no catalog, no switch on method names. This is why adding a new component requires zero runtime changes.
