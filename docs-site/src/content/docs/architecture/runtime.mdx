The runtime is a plan interpreter. It reads the JSON plan that C# builders produce and executes the described behavior. It does not invent behavior — it follows instructions.

## Module structure

```
Scripts/
├── root.ts                          entry point — discovers [data-reactive-plan], calls boot()
├── lifecycle/                       boot, merge, enrichment
│   ├── boot.ts                      two-phase boot — wire listeners, then execute DomReady
│   ├── enrichment.ts                resolves component refs to IDs, vendors, readExprs
│   ├── merge-plan.ts                composes partial plans (ResolvePlan merges into parent)
│   └── walk-reactions.ts            traverses reaction tree for validation + live-clear wiring
├── execution/                       where behavior happens
│   ├── execute.ts                   reaction dispatcher — sequential, conditional, http, parallel
│   ├── commands.ts                  command dispatcher — dispatch, mutate-element, mutate-event, into
│   ├── element.ts                   bracket notation — root[prop] = val, root[method](args)
│   ├── trigger.ts                   wires dom-ready, custom-event, component-event listeners
│   ├── gather.ts                    collects component + event values for HTTP requests
│   ├── http.ts                      builds and executes fetch(), routes responses
│   ├── pipeline.ts                  orchestrates: pre-fetch → validate → request → parallel
│   └── inject.ts                    injects HTML into containers, merges embedded plans
├── resolution/                      where values come from
│   ├── resolver.ts                  walks BindSource — event paths or component reads
│   └── component.ts                 vendor root resolution (see Component Model)
├── conditions/                      evaluates guards — value, all, any, not, confirm
├── validation/                      client-side validation from plan descriptors
├── components/                      component-specific side effects (drawer, loader, confirm)
├── core/                            walk.ts, coerce.ts, trace.ts, assert-never.ts
└── types/                           TypeScript interfaces mirroring the JSON plan schema
```

## Boot — from page load to execution

```d2
direction: right

discover: "root.ts\nFind all\n[data-reactive-plan]" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 18
}
parse: "Parse JSON\nplan + components" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 18
}
enrich: "Enrichment\nResolve component refs\nto IDs, vendors, readExprs" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 18
}
phase1: "Two-Phase Boot\nPhase 1: Wire all listeners" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 18
}
phase2: "Two-Phase Boot\nPhase 2: Execute DomReady" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 18
}

discover -> parse {
  style.stroke: "#4285f4"
}
parse -> enrich {
  style.stroke: "#f9ab00"
}
enrich -> phase1 {
  style.stroke: "#f9ab00"
}
phase1 -> phase2: listeners ready {
  style.stroke: "#4285f4"
}
```

**Two-phase boot** guarantees that event chains work regardless of declaration order. Phase 1 wires all CustomEvent and component-event listeners. Phase 2 runs DomReady reactions — which may dispatch events that Phase 1 listeners catch. Without this separation, a DomReady dispatch would fire before the listener exists and the event would vanish silently. Chains of any depth work because `dispatch` fires synchronous `CustomEvent`s on `document`.

At this point, the page is live. Listeners are attached. The runtime is waiting. What happens when the user actually does something?

## Command execution

Three dispatch layers fire in sequence — each one narrowing from abstract to concrete. A trigger fires a reaction. The reaction contains commands. Each command applies a mutation. This layering is what makes the runtime extensible without becoming complex.

### Layer 1 — Reaction dispatch

When a trigger fires, `execute.ts` reads `reaction.kind` and picks the execution strategy:

```d2
direction: right

trigger: "Trigger fires\nchange event on checkbox" {
  shape: rectangle
  style.fill: "#fce8e6"
  style.stroke: "#ea4335"
  style.font-size: 18
}
sequential: "sequential\nRun commands in order" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 18
}
conditional: "conditional\nEvaluate guards\nPick first matching branch" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 18
}
http: "http\nValidate → Gather →\nFetch → Route response" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 18
}
parallel: "parallel-http\nPromise.all on branches\nthen OnAllSettled" {
  shape: rectangle
  style.fill: "#e8f0fe"
  style.stroke: "#4285f4"
  style.font-size: 18
}

trigger -> sequential: kind {
  style.stroke: "#ea4335"
}
trigger -> conditional: kind {
  style.stroke: "#ea4335"
}
trigger -> http: kind {
  style.stroke: "#ea4335"
}
trigger -> parallel: kind {
  style.stroke: "#ea4335"
}
```

A conditional reaction can contain an HTTP reaction inside its branch. An HTTP response handler can contain a conditional. The reaction tree is recursive — but at each node, the dispatch is a flat switch on `kind`.

### Layer 2 — Command dispatch

Inside a reaction, `commands.ts` iterates the command list and dispatches each one by `command.kind`:

| Command kind | What it does | Module |
|-------------|-------------|--------|
| `dispatch` | Fires `CustomEvent` on `document` | inline |
| `mutate-element` | Sets a property or calls a method | `element.ts` |
| `mutate-event` | Modifies event args in execution context | inline |
| `validation-errors` | Displays server-side errors at form fields | `validation/` |
| `into` | Injects response HTML into container | `inject.ts` |

Each command can carry an optional `when` guard — evaluated before the command runs. If the guard fails, that command is skipped but the rest of the pipeline continues.

### Layer 3 — Mutation execution

`element.ts` is where the plan meets the DOM. It has exactly two primitives:

```d2
direction: right

resolve: "Resolve root\nnative → el\nfusion → el.ej2_instances[0]" {
  shape: rectangle
  style.fill: "#fef7e0"
  style.stroke: "#f9ab00"
  style.font-size: 18
}
setprop: "set-prop\nroot[prop] = val" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 18
}
call: "call\nroot[method](args)" {
  shape: rectangle
  style.fill: "#e6f4ea"
  style.stroke: "#34a853"
  style.font-size: 18
}

resolve -> setprop: mutation.kind {
  style.stroke: "#f9ab00"
}
resolve -> call: mutation.kind {
  style.stroke: "#f9ab00"
}
```

`root[prop] = val` handles `SetText`, `SetValue`, `SetChecked`. `root[method](args)` handles `AddClass`, `DataBind`, `ShowPopup`. The plan carries the property or method name — the runtime executes it verbatim via bracket notation. No catalog of operations, no switch on method names.

This is why adding a new component method requires zero runtime changes. The C# extension method puts the name in the plan. The runtime applies it.

How does `resolveRoot` know which root to return? That's the vendor resolution pattern — covered in [Component Model](/architecture/component-model/).
